Core Features
1. Multi-File Excel Uploader
Drag-and-drop file upload interface supporting multiple XLSX/XLS files simultaneously, with a 200MB per-file limit. A paginated file manager displays all uploaded files (e.g., TB_GAL_06.xls, TB_GAL_07.xls, TB_GAL_08.xls) with per-file removal. Pagination handles large file sets cleanly ("Showing page 1 of 3").
2. GL ID Tag Filter
A multi-tag input field allows finance analysts to enter specific GL account IDs as removable chips/tags. The selected IDs (e.g., 1010200008, 1031020132, 2050601391, 4000050003, 6001001004) drive all downstream views — the table, KPI card, and both charts — enabling focused account-level drill-down without page reloads.
3. Filtered GL ID Table
A structured data table showing month-by-month balances (Apr–Nov) and FTM (First-to-Month cumulative) columns per GL account. Cells are conditionally formatted in red/green based on movement direction, enabling instant visual identification of variance. Accounts shown include VEHICLES, CA-IDFC Bank, CGST RCM, NOTICE PAY, and ALLOWANCE CHILD EDU.
4. KPI Variance Card with Account Selector
A prominent KPI metric card displaying total variance (e.g., ₹2,531,548 with ↑114% change badge) for the selected GL account. A dropdown lets users switch the KPI context to any filtered GL account instantly.
5. Accumulated Balances Over Time (Line Chart)
A multi-series line chart tracking the cumulative balance trajectory of each GL account from April to November. Allows identification of accounts with steep growth curves (e.g., ALLOWANCE CHILD EDU surging to ₹2.8M) versus flat or declining accounts.
6. Month-to-Month Change Chart (Bar Chart)
A grouped bar chart showing the period-on-period delta for all accounts per month range. Positive and negative bars are color-coded, making seasonal spikes and reversals immediately visible.
7. Trail Balance Sheet Heatmap
A full GL-level heatmap grid showing all accounts with Jul–Nov absolute values and all FTM columns, alongside computed Variance and Change% columns. Color intensity scales with magnitude — deep red for large negatives, deep green for large positives — across thousands of GL rows. Variance percentages range from -114% to +50,029%, captured cleanly in the grid.

Pipeline Architecture — 4 Stages
Stage 01 · Source Files
Three input files feed the pipeline, each with a defined role:
openDump.xlsx (Open AR Dump): Contains Customer ID, Customer Name, Reference Key 1 (mapped to Invoice), and Amount in local currency. This is the AR system's outstanding balance export.
cm.xlsx (Customer Master): Contains Customer ID and Name. Used as the authoritative source for customer name enrichment via lookup join.
Extracted_Invoices_Consolidated.xlsx (Invoice Extract): Contains Invoice Number and Amount columns. The amount column is auto-detected by name pattern — making the pipeline resilient to minor schema variations across exports.
Stage 02 · Transformations
Each source undergoes independent cleaning before the merge:
Open AR Dump — Clean + Group: Strips .0 suffixes from invoice numbers, uppercases all invoice strings for consistent key matching, groups by Invoice and sums amounts (many-to-one), and flags duplicate invoice entries (count &gt; 1) for downstream flagging.
Customer Master — Lookup Join: Performs a left join onto the grouped Open Dump dataframe. Maps Customer → Customer_ID and Name → Customer_Name_Master for enrichment.
Invoice Extract — Clean + Detect: Applies the same .0 strip and uppercase normalization. Auto-detects the amount column by scanning column names, then parses amounts to float.
Stage 03 · Merge Logic
Invoice Extract serves as the base table. A LEFT JOIN on Invoice key brings in OpenDump Grouped + Customer Master. This ensures all invoices from the extract appear in output, even if missing from the AR dump (NOT_IN_OPENDUMP status).

Core Technology Deep-Dive
1. Tavily API — Real-Time AI Web Research
Tavily is an AI-optimised search API built specifically for LLM agents. Unlike standard search engines, it returns clean, structured, LLM-ready content rather than raw HTML — making it ideal for agentic research pipelines.
Purpose in this project: Used to fetch the latest AI news, research papers, product launches, policy developments, and business updates from across the web in real time — every time the agent runs.
How it works: The agent sends targeted search queries to Tavily (e.g., 'latest AI model releases this week', 'AI policy news 2026') and receives back structured article data: title, URL, summary, and source domain.
Why Tavily over Google Search API: Tavily is purpose-built for agents — it understands semantic intent, filters low-quality results, and returns content that is directly digestible by an LLM without additional scraping or parsing.
Result: Each newsletter section (Headlines, Research, Business, etc.) is populated with fresh, relevant, real-world content discovered autonomously — not hardcoded or templated.
2. Groq API — Ultra-Fast LLM Inference
Groq is a high-performance AI inference platform running open-source LLMs (such as LLaMA 3, Mixtral) on custom LPU (Language Processing Unit) hardware — delivering significantly faster inference than GPU-based providers.
Purpose in this project: Used to summarise raw article content fetched by Tavily into clean, readable newsletter paragraphs — and to write the Takeaways section with synthesised insights across all stories.
Why Groq over OpenAI / Claude for this task: Groq's inference speed is critical for an automated pipeline. Since the agent processes 10-20 articles per run, fast inference keeps total runtime under 30 seconds — making it viable for scheduled execution without timeout issues.
Prompt design: Each article is summarised using a system prompt that enforces newsletter tone: concise, non-technical, insight-first writing. The Takeaways section uses a multi-article synthesis prompt that extracts 3-5 actionable insights for the reader.
Model used: LLaMA 3 70B via Groq's API — balancing quality and speed for newsletter-grade summarisation.
3. Cron Jobs — Fully Automated Scheduled Delivery
The entire pipeline is orchestrated by a cron job — a time-based Unix scheduler that triggers the agent script at a fixed time every week, requiring no manual intervention.
Schedule: The agent runs every Wednesday. The live email timestamped Wed, March 11, 2026 at 3:03 PM confirms the cron is firing correctly on schedule.
Cron expression used: 0 9 * * 3 — triggers at 9:00 AM UTC every Wednesday (3:03 PM IST accounting for timezone offset). The cron runs on a cloud server (e.g., Linux VM / EC2) that is always on.
Pipeline flow on trigger: Cron fires → Python script starts → Tavily fetches articles per category → Groq summarises each article → HTML email template is assembled → SMTP sends email via Gmail → Job completes.
Error handling: The script includes try-except blocks per section so a single failing API call does not abort the entire newsletter — the section is skipped or filled with a fallback message.
Infrastructure: Hosted on a Linux server. The cron entry is registered via crontab -e. Environment variables (API keys for Tavily, Groq, Gmail SMTP) are stored in a .env file loaded at runtime.

🔄 Power BI Automation Process
Data Source – Email
Every morning, data files are received through email.
Automation using Microsoft Power Automate
A flow monitors the mailbox automatically.
When a new email with the data file arrives:
The flow detects the attachment.
Downloads the file automatically.
Stores it in a shared folder (for example SharePoint/OneDrive/network drive).
Data Visualization using Microsoft Power BI
The Power BI report is connected to the shared folder where the file is stored.
A scheduled refresh is set for 10:00 AM every day.
End Result
Every day:
Email arrives → Flow saves the file → Power BI refreshes.
The dashboard shows latest (almost live) data automatically without manual work.
💡 How You Can Explain It in an Interview
You could say something like this:
“I developed an automation solution using Power Automate and Power BI. The data files are received via email every morning. I created a Power Automate flow that automatically detects the email attachment, downloads the file, and saves it to a shared folder. The Power BI dataset is connected to that folder, and I scheduled a daily refresh at 10 AM. This ensures the dashboard is automatically updated with the latest data every day without manual intervention.”
